This instruction set is for creating a set of help documents in markdown format that describe canlab core tools. Format all documents so they will render well when used on the web at canlab.github.io.

In this task, we will create a set of .md pages for selected object methods and key functions There are 3 phases: 1: Build code maps.  2: Build .md markdown files for each method, including the associated code map. 3: Modify object class methods .md pages to link to the new pages. 

Phase 1
Read the file canlab_template_codemap.pptx.  This is a code map that specifies the flow of data through a set of functions. I will give you a list of functions below.
For each function in the list below, create a native powerpoint file with editable powerpoint-native objects, including text boxes and arrows, that provide a graphical code map of the function and its subfunctions, inputs, outputs, and key variables. Use pptxgenjs, or a preferred skill if there is a better option available. Use the slide in template_codemap.pptx as a template, keeping the font, color scheme, and style consistent with the template.  Name the new pptx file with the object class name and method name followed by _codemap, i.e., “<class>_<method>_codemap.pptx”, for example ”fmri_data_regress_codemap.pptx”.  If a .pptx file of the name already exists, read it and consider whether it needs updates, and make any updates needed instead of recreating it from scratch. If you create or modify the file, save the pptx file in docs/code_maps_pptx/.  Then save a .png file of the new codemap with the same name but ending in .png in the folder docs/code_maps_png/.   Crop out extra white space from the .png.

Layout rules learned from the first pass (avoid these problems in future codemaps)
==================================================================================

The Use/Notes column on the left is reserved space.
- The "Use:" bullet block lives at roughly x=0.25 to x=2.85, y≈0.55 to y≈3.40.
- The "Notes:" bullet block lives at roughly x=0.26 to x=2.36, y≈3.50 to y≈5.10.
- The "Legend" lives at roughly x=0.30 to x=2.30, y≈5.30 to y≈6.50.
- Therefore: NO box, arrow, annotation, or input/output node may be placed
  at x < 2.85 with y > 0.85. Output rows in particular must start at x ≥ 2.90.
- The slide's right edge is at x = 13.33"; keep all boxes inside x ≤ 13.10.

Use/Notes content discipline.
- Use bullets: max 3 short bullets. Long fourth/fifth bullets push the column
  down into the Notes header and overlap.
- Notes bullets: max 2 bullets. Long single bullets (more than ~25 words)
  wrap into 4+ rendered lines and bleed into the Legend area.
- If a function legitimately needs more guidance than that, put it in the
  per-function .md page, not in the codemap.

Title and box-label fitting.
- Long titles (>45 chars in the title string, including the "CANlab code map: "
  prefix) auto-shrink in the helper. You do not need to do anything special.
- For long function names inside a box (e.g. annotate_binary_results_map( )),
  widen the box (box_w=2.85 instead of the default 1.85) — do NOT let the
  text wrap mid-word. A wrapped function name like "annotate_binary_results_m\nap( )"
  is the wrong fix; widen the box.
- Avoid two-line node labels longer than 18 characters per line — they hit
  the box border in the cropped png.

Arrow rules — the most common source of ugly codemaps.
1. Connect the bottom of the LAST spine node to outputs, never a middle one.
   If you draw connect_line(simfn, output) but there is also a "plot" node
   between simfn and output, the line goes through the plot box. Always
   connect from the closest spine node above the outputs.
2. Avoid fan-in arrows from a 2-row grid into a single sink. Each arrow from
   row 1 of the grid passes through row 2 boxes. Instead: collapse the grid
   into a single grouped panel ("internal extractors:\n• ...\n• ...") and
   draw ONE arrow from the panel to the sink.
3. Don't draw a redundant skip-connection arrow that bypasses spine nodes.
   E.g. spine: rob → diag → OUT. Drawing connect_line(rob, OUT) in addition
   to connect_line(diag, OUT) makes the rob→OUT line cross the diag box.
   The implicit chain through diag is enough.
4. For diagonal arrows that would cross a third box, use src_side/dst_side
   anchors that go around it (e.g., bottomleft→topright instead of
   bottom→top), or use kind="bent" for an L-shaped elbow line.
5. Inputs and side-branches go on the RIGHT of the spine (x > spine_x + 1.85).
   Putting them on the left puts them in the Use column.

Object types panel placement.
- Place it in a clear corner — usually upper-right (x = spine_x + 3.5, y = y0)
  or lower-right (x = spine_x + 3.5, y = out_y + 1.0).
- Never place it at the same x as an input box or output box at the same y;
  the panel's heading "Object types:" will visually merge with the box label.
- For 4+ entries, give it width=2.40 to 2.80; otherwise long names like
  "image_vector" or "MATLAB table" wrap.

Output row placement.
- Standard output row: y = y0 + (n_spine_nodes) * gap + 0.10–0.20.
- Standard output positions: 5 outputs → x = 2.90 + i * 2.05 (col_w 1.95,
  col_gap 0.10); 4 outputs → x = 2.90 + i * 2.55; 3 outputs → x = 2.90, 5.95,
  9.00 (each w = 2.80–3.20). These leave the Use/Notes column clear and stay
  inside the slide bounds.

Render-and-check loop.
- After writing or editing a spec, run `PYTHONPATH=. python3 build_all.py
  <name>` from docs/_codemap_tools/ and READ the resulting png. Visual
  inspection catches all of the above issues; static review of the spec does
  not.

Phase 2.
For each method or stand-alone function you have created a code map for, create a .md file that describes the function. Start by copying the help from the Matlab help block at the beginning of the method/function file and modifying it to look good in markdown. Insert a link to the code map after the help block. Save the file in docs/individual_functions/ with the naming convention <class>_<method>.md, e.g., fmri_data_regress.md, or <function_name>.md for stand-alone functions. 

Phase 3.
Then, modify the appropriate <class>_methods.md file (e.g., fmri_data_methods.md for fmri_data methods) to replace the method name in the table with a hyperlink to the new page for the corresponding method. For methods associated with a child class of the @image_vector class, modify the <class>_methods.md file for both the image_vector class and the child class (e.g., fmri_data, statistic_image, etc.).
