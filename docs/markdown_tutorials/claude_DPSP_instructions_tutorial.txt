1. Load data.  Explanation to draw from in tutorial: The data is first-level condition maps from single subjects, averaging over runs. We define condition maps as task - baseline, where the baseline is the implicit baseline captured by the first-level model intercept. So we have, e.g., single_subject_images_hot, which stores a Hot - Baseline activity map for each participant. Same for the other 3 conditions. Because we have jitter between events, we can estimate the baseline, and thus task - baseline images are stable. In many tasks, baseline is rest. but here it's a visuospatial task, chosen by design because rest could invite periods of rumination about past events (like the picture of one's ex).

Each fmri_data object has subjects in the same order and a .metadata table field with a table object. Explain this briefly.  The metadata table has a row corresponding to each image in the object.  Here it contains a subject ID (one participant per row; subj_id), a condition label ("Condition"), which will make it easy to keep track of conditions later when we stack them for SVM/classifier analysis, and an assignment to cross-validation set or final holdout set ("Orig_partition"), which is convenient for splitting data later in the demo. it is a good idea to have independent test datasets that you test only once on the final model you develop (the "Test" partition here) if you can, to evaluate accuracy, though you need at least a moderately sized study in terms of numbers of participants to do this meaningfully. n = 60 is probably a reasonable minimum.
