0. EXECUTIVE SUMMARY
===============================================================================

The current @predictive_model class is a container — it has properties and a
struct-copy constructor, but every analytic method (`train`, `test`, `crossval`,
`bootstrap`, `permutation_test`, `select_features`, ...) is a stub that errors.
Two visualization methods are implemented (`confusion_matrix`,
`plot_predicted_vs_observed`). It is consumed by xval_SVM/xval_SVR only via
the struct-to-object conversion at the very end of those wrappers.

The recommendation is to evolve this from a passive results container into an
active model object with a scikit-learn-shaped fit/predict/score surface, while
keeping it as the canonical output type for fmri_data.predict, xval_SVM,
xval_SVR, and future wrappers. The shape that maps cleanly onto MATLAB is:

  - hyperparameters (algorithm name, kernel, lambda, nfolds, modeloptions, ...)
    live in a clearly separated property block, set by constructor
  - fitted artifacts (Beta, intercept, ml_model, yfit, cv predictions, bootstrap
    stats) live in a second property block, written only by fit / crossval /
    bootstrap methods
  - the trained MATLAB model object (ClassificationSVM, RegressionLinear,
    TreeBagger, ...) is stored verbatim and predict/loss/score delegate to it
  - subject grouping (the `id` argument that xval_SVM/SVR thread through today)
    becomes a first-class CV concept via a small set of splitter objects
  - cross-validation, hyperparameter search, bootstrap, and permutation are
    composable methods that all consume and return predictive_model objects

This gives users a clean Python-style "fit-predict-score-cross_validate" mental
model, with the neuroimaging-specific additions CanlabCore already cares about
(grouped CV, voxelwise weight bootstrapping, FDR-thresholded weight maps,
within-person effect sizes, image-aware predict).

1. CURRENT STATE OF THE CODE
===============================================================================

@predictive_model/
  predictive_model.m          -- classdef, ~50 properties, struct-copy ctor,
                                 validate_object method, stubbed analytic methods
  confusion_matrix.m          -- implemented (uses MATLAB confusionmat/chart)
  plot_predicted_vs_observed.m-- implemented (scatter for regression, violin for
                                 binary classification via barplot_columns)
  crossval.m                  -- EMPTY FILE (1-line stub `function crossval(...)`)

Consumers / producers today:

  Statistics_tools/Cross_validated_Regression/xval_SVM.m
  Statistics_tools/Cross_validated_Regression/xval_SVR.m
      Both run cross-validation internally (their own fold loops, their own
      Bayesian hyperparameter optimization, their own bootstrap), assemble a
      flat MATLAB struct `S`, and then call `S = predictive_model(S)` at the
      end. predictive_model's only job in this pipeline is shape-checking and
      a few legacy-field translations (SVMModel/SVRModel -> ClassificationModel;
      classification_d_singleinterval / regression_d_singleinterval -> d_singleinterval).

  @fmri_data/predict.m
      Does NOT currently return a predictive_model. Returns `[cverr, stats, optout]`
      where stats is a flat struct with very similar fields. Algorithm dispatch
      is keyword-based: 'cv_pcr' (default), 'cv_lassopcr', 'cv_lassopcrmatlab',
      'cv_svm' (Spider), 'cv_svr' (Spider), 'cv_pls', 'cv_multregress',
      'cv_univregress', 'cv_multilevel_glm', 'cv_mlpcr'. Built-in bootstrap of
      voxel weights; built-in stratified k-fold; choice of error metric;
      hv-block and rolling-window CV for time series.

Observations:

  - There are at least three CV/bootstrap engines in CanlabCore that re-implement
    the same loop with small variations: fmri_data.predict (algorithm-name
    dispatch), xval_SVM (full pipeline), xval_SVR (full pipeline). Each has
    its own bug history (see the long programmer-notes block at the bottom of
    predict.m). Consolidating these onto one predictive_model engine is the
    single biggest payoff of this redesign.
  - validate_object in predictive_model.m has a known issue with the
    `validateModeloptionsCell` predicate (it doesn't actually short-circuit
    correctly on key-value cells; Byeol added a comment, June 2025). Worth
    fixing as part of the refactor.
  - The "forthcoming" methods listed in the docstring are a useful checklist of
    what the class is *intended* to do; the redesign should deliver on that
    list rather than re-litigate it.

2. REFERENCE DESIGNS
===============================================================================

------------------- 2a. scikit-learn estimator API ----------------------------

Why look here: sklearn is the de facto reference design for predictive-model
APIs in machine learning. CanlabCore users routinely move back and forth to
Python; the closer the mental model, the lower the cognitive tax.

The five sklearn ideas worth borrowing:

  1. fit/predict/score contract.
     Every estimator implements fit(X, y) -> self, predict(X) -> yhat, and
     score(X, y) -> float (higher-is-better). Classifiers additionally
     implement decision_function(X) and predict_proba(X); regressors don't.
     This is a small surface, and learning it transfers across every model
     type.

  2. Strict separation of hyperparameters from fitted state.
     Constructor stores hyperparameters only, no logic. Fitted artifacts get
     a trailing `_` (coef_, intercept_, classes_, n_features_in_). The
     trailing-`_` convention is ugly in MATLAB, but the INVARIANT -- two
     disjoint sets of properties, with `isFitted` testable -- is golden.
     It makes clone(), refit semantics, and serialization unambiguous.

  3. Splitter objects with .split(X, y, groups).
     KFold, StratifiedKFold, GroupKFold, StratifiedGroupKFold,
     LeaveOneGroupOut, RepeatedKFold, ShuffleSplit. All have the same
     iterator API. `groups` is a first-class argument -- exactly CanlabCore's
     `id` vector. Borrow the name and the convention.

  4. Composable model selection.
     cross_validate(estimator, X, y, cv, scoring) returns a per-fold struct
     with test_score, train_score, fit_time, estimator. GridSearchCV wraps
     any estimator and exposes the same fit/predict surface, so nested CV
     is just cross_validate(GridSearchCV(est, grid, cv=inner), ..., cv=outer)
     with no new abstraction. This composability is what makes sklearn
     extensible; CanlabCore's xval_SVM hard-codes the nested CV inside the
     function and pays for that with bugs and inflexibility.

  5. Scorer objects with greater_is_better flag.
     Don't bake sign into metric names ("neg_mean_squared_error" is widely
     regretted). Store an explicit greater_is_better flag so a single
     argmax loop works for accuracy, MSE, AUC, etc.

Two sklearn ideas to deliberately diverge from:

  1. trailing-`_` attribute names. Carry the invariant via a properties-block
     split (`properties (SetAccess = public)` for hyperparameters,
     `properties (SetAccess = protected)` for fitted state) instead.

  2. set_params() mutating in place while returning self. MATLAB's existing
     CanlabCore classes are VALUE classes, so set_params should return a new
     object. Document this loudly for users coming from Python.

One sklearn feature worth replicating in spirit even if not in code:
metadata routing (post-1.4). The historical problem -- "how do I pass `id`
through nested CV?" -- is exactly what CanlabCore solves ad hoc today. The
minimum-viable version is a `params` struct passed to crossval / grid_search
whose fields (`groups`, `sample_weight`) are dispatched to the consumers
that have declared an interest. Don't replicate the full machinery; do
adopt the contract.

------------------- 2b. MATLAB Statistics & ML Toolbox -------------------------

Why look here: this is what we actually wrap. The toolbox is already a
clean, uniform API; the worst thing we can do is reinvent its predict()
poorly.

The shared method surface across ClassificationSVM, ClassificationLinear,
ClassificationKNN, ClassificationTree, ClassificationEnsemble, TreeBagger,
ClassificationECOC, ClassificationNeuralNetwork, ClassificationGAM,
ClassificationNaiveBayes, and their Regression* counterparts:

  predict(mdl, X)             - labels or values; second output is score/probability
  loss(mdl, X, Y)             - generalization error (LossFun selects metric)
  margin(mdl, X, Y)           - classification only
  edge(mdl, X, Y)             - classification only; weighted-mean margin
  resubPredict, resubLoss     - in-sample (no CV)
  crossval(mdl)               - returns Classification/Regression PartitionedModel
                                with .Trained (cell of per-fold models),
                                .kfoldPredict, .kfoldLoss, .kfoldMargin, .kfoldfun
  oobPredict, oobLoss         - bagged ensembles only
  compact(mdl)                - strip training data; ESSENTIAL for fMRI-scale weights

Plus a uniform OptimizeHyperparameters name/value across every fit*
function, returning a BayesianOptimization object stored at
mdl.HyperparameterOptimizationResults.

Two gaps that motivate keeping the CanlabCore wrapper layer:

  1. No first-class grouped CV. cvpartition has KFold, Holdout, Leaveout,
     Stratify, and a 'CustomPartition' escape hatch, but no GroupKFold or
     LeaveOneGroupOut. CanlabCore must continue to hand-roll these from the
     `id` argument (this is what xval_SVM does today via stratification on
     unique-id-by-outcome).
  2. fitclinear / fitrlinear handle the high-dimensional case dramatically
     better than fitcsvm / fitrsvm but expose a slightly different API
     (Beta is [p x numel(Lambda)] for a sweep; no KernelParameters; limited
     OptimizeHyperparameters). The wrapper needs an explicit
     `highdimensional` flag (which xval_SVM already has) that picks the
     right MATLAB fit function.

What the wrapper SHOULD add on top of the MATLAB objects:

  - subject-grouped CV (`id`) via a GroupKFold-equivalent splitter
  - bootstrap of weight maps (voxelwise SE, Z, p, FDR thresholding) ->
    output usable as a statistic_image
  - within-person effect sizes (paired forced-choice accuracy,
    classification_d_within, scorediff) for repeated-measures designs
  - image-space awareness: predict() should accept an fmri_data /
    image_vector / filename, dispatch via isa(), and resample to the model's
    feature space if needed
  - permutation tests (null distribution under shuffled Y; not built into
    MATLAB)
  - calibration as a meta-estimator (Platt scaling done once, principled,
    not per-fold ad hoc)
  - neuroimaging-aware visualization (montage of weight_obj, ROC with
    subject-clustered CIs, etc.)
  - history / removed_voxels bookkeeping consistent with image_vector

What the wrapper should NOT reimplement: predict, loss, margin, edge,
kfoldPredict, oobPredict, OptimizeHyperparameters. These already work on
the trained MATLAB object stored at obj.ml_model; the wrapper just
delegates.

===============================================================================
3. DESIGN GOALS FOR CanlabCore predictive_model
===============================================================================

G1. ONE OBJECT, ONE INTERFACE.
    predictive_model is the canonical output of every CanlabCore predictive-
    modeling entry point (fmri_data.predict, xval_SVM, xval_SVR, and future
    wrappers). Downstream code (plotting, montaging, region tables) sees one
    type, not three.

G2. FIT / PREDICT / SCORE / CROSSVAL / BOOTSTRAP AS METHODS, NOT STUBS.
    The "forthcoming" methods become real. The constructor still accepts a
    struct (back-compat), but the primary path is:
        m = predictive_model('algorithm','svm','modeloptions',{...});
        m = fit(m, X, Y, 'id', id);
        [yhat, score] = predict(m, Xnew);
        cv = crossval(m, X, Y, 'id', id, 'cv', group_kfold(10));
        bm = bootstrap(m, X, Y, 'nboot', 1000);
        pm = permutation_test(m, X, Y, 'nperm', 5000);

G3. HYPERPARAMETERS AND FITTED STATE ARE STRICTLY DISJOINT.
    Two property blocks. isFitted(obj) tests for any populated fitted-state
    field. clone(obj) returns a fresh unfitted copy with identical
    hyperparameters. This buys us bug-free per-fold cloning and clean
    serialization.

G4. THE TRAINED MATLAB OBJECT IS STORED VERBATIM AND CALLED DIRECTLY.
    obj.ml_model is the underlying ClassificationSVM / RegressionLinear /
    TreeBagger / ... Predict, loss, etc. delegate. We do not wrap the
    MATLAB API; we use it.

G5. SUBJECT GROUPING IS A FIRST-CLASS CV CONCEPT.
    A small set of splitter objects (kfold, stratified_kfold, group_kfold,
    stratified_group_kfold, leave_one_group_out, repeated_group_kfold)
    with a uniform .split(X, Y, groups) -> {trIdx, teIdx} iterator API.
    `id` becomes `groups` passed into crossval, NOT a model property.

G6. NESTED CV IS COMPOSABLE.
    A predictive_model whose body is itself a tuner (grid_search /
    bayes_search) is just another predictive_model. crossval works on it
    without special-casing. This is how we get rid of the nested-CV
    duplication across xval_SVM, xval_SVR, and predict.

G7. NEUROIMAGING-AWARENESS IS A METHOD CONCERN, NOT A CLASS CONCERN.
    The core predictive_model takes numeric (X, Y, groups). A thin layer
    on top -- e.g. fmri_data.fit_predictive_model -- handles the conversion
    from fmri_data into the (X, Y, groups) triple, and the inverse
    conversion from obj.Beta into an fmri_data / statistic_image weight map.
    This keeps the core class testable without SPM and without fMRI data.

G8. BACKWARDS COMPATIBLE WITH EXISTING xval_SVM/SVR STRUCT OUTPUTS.
    predictive_model(struct) keeps working. Legacy field translations
    (SVMModel -> ml_model, classification_d_singleinterval -> d_singleinterval)
    keep working. We are extending, not breaking.

G9. NO HALF-FINISHED METHODS.
    Every shipped method is either real or absent. The placeholder pattern
    of "forthcoming, errors when called" is removed -- those methods stay
    out of the class until they are implemented (this is the project
    convention from CLAUDE.md: no half-finished implementations).

===============================================================================
4. PROPOSED CLASS ARCHITECTURE
===============================================================================

----------- 4a. Single class vs. subclass-per-type ----------------------------

RECOMMENDATION: single class with `task` property that is 'classification' or
'regression', plus is_classifier(obj) / is_regressor(obj) predicates.

Why not subclass:
  - MATLAB value-class inheritance is awkward; users have to remember which
    constructor to call.
  - The vast majority of methods (fit, predict, score, crossval, bootstrap,
    permutation_test, clone, plot, save, load, compact) are identical
    between classification and regression. The differences (predict_proba,
    margin, confusion_matrix vs scatter) are cleanly handled by branching
    on `task`.
  - Single class plays nicely with the existing struct-copy constructor.

If we DO want subclasses later, they should be predictive_classifier and
predictive_regressor inheriting from predictive_model, but the v1 design
is flat.

----------- 4b. Properties: categorised layout (IMPLEMENTED Phase 0) ----------

After surveying the union of fields across the 11 Category-A xval_*
wrappers (~50 distinct field names), a flat property layout was rejected
as unworkable (the class would have ballooned to ~85 flat properties with
no method support, violating G9). The implemented layout instead groups
fitted state into ~13 named sub-struct properties, with the legacy flat
names exposed as Dependent (read-only) aliases so existing code that does
`pm.w` / `pm.yfit` / etc. keeps working.

HYPERPARAMETERS (top-level, set by user/constructor):

    properties  % HYPERPARAMETERS
        algorithm       % string: 'svm','svr','linear_svm','lassopcr',
                        %         'pcr','ridge','lda','pls','logistic',
                        %         'tree','rf','knn','gp','nnet','ecoc',
                        %         'gam','naive_bayes','custom'
        task            % 'classification' | 'regression'
        modeloptions    % cell of name/value pairs forwarded to MATLAB fit fn
        random_state    % rng seed
        standardize     % logical
        use_parallel    % logical
        cv              % splitter object (Phase 1)
        scorer          % scorer object (Phase 1)
        nboot           % bootstrap samples
        nperm           % permutation samples
        do_calibrate    % logical (Platt-style calibration)
        Y_name, X_name, class_labels
    end

FITTED STATE (categorised sub-struct properties + a few top-level fields):

    properties (SetAccess = protected)
        inputs               % .Y .id .Y_orig .INPUTS
        cv_partition         % .trIdx .teIdx .nfolds .fold_modeloptions
                             %   .hyperparams_by_fold .cvpartition
        fitted_values        % .yfit .dist_from_hyperplane_xval
                             %   .class_probability_xval .scores_within_id
                             %   .Y_within_id .scorediff .subjfit
                             %   .high_vs_low_scores_within_id
                             %   .predictions .trueLabels
        weights              % .w (= Beta) .mean_vox_weights .vox_weights
                             %   .VOXWEIGHTS .my_intercepts .subjbetas
                             %   .weight_obj  (statistic_image when available)
        weight_stats         % .boot_w .boot_w_ste .boot_w_mean .wZ .wP
                             %   .wP_fdr_thr .boot_w_fdrsig .w_thresh_fdr
        error_metrics        % .crossval_accuracy .crossval_accuracy_within
                             %   .d_singleinterval .d_within
                             %   .prediction_outcome_r .cverr .mse .rmse
                             %   .meanabserr .r_squared .var_full .var_null
                             %   .var_reduction .pred_err .pred_err_null
                             %   .devs_from_full_model
                             %   .devs_from_mean_only_model .r_each_subject
                             %   .accuracy .overallAccuracy .err .phi
        descriptions         % .accfun_descrip .cverrfun
                             %   .crossval_accuracy_descrip
                             %   .prediction_outcome_r_descrip
                             %   .pred_err_descrip .var_reduction_descrip
                             %   .classification_d_within_descrip
                             %   .note .r_each_subject_note
                             %   .r_each_subject_note2 .cvoutput_descrip
                             %   .error_type .function_call .function_handle
                             %   .algorithm_name
        ml_model             % the trained MATLAB model object (verbatim).
                             %   Aliases: SVMModel, SVRModel, ClassificationModel
                             %   all land here via the routing table.
        fold_models          % cell of per-fold trained MATLAB models.
                             %   Alias: "models".
        bootstrap_results    % full bootstrap output struct (e.g., the
                             %   .bootstrap sub-struct produced by
                             %   xval_lasso_brain). Aliases: bootstrap, WTS,
                             %   boot_weights.
        permutation_results  % full permutation output struct
        diagnostics          % .ROC_forced_choice .ROC_single_interval
                             %   .mult_obs_within_person .all_reg_hyperparams
        cross_classify       % cross-classification output (.stats1 .stats2
                             %   .test_results .crosstestfit .cvoutput
                             %   .test_Y .all)
        legacy_extras        % catch-all struct for fields without a
                             %   categorised home (covs, data, full_model,
                             %   inputOptions, ...) — stored verbatim so
                             %   nothing is lost during struct -> object.
        accfun               % top-level: objective function handle
        history              % cell of strings (image_vector convention)
        removed_voxels       % logical (image_vector convention)
    end

DEPENDENT READ-ONLY LEGACY ALIASES (top-level, forwarding getters):

    properties (Dependent, SetAccess = private)
        Y, id, yfit, w, boot_w, boot_w_ste, boot_w_mean, wZ, wP,
        wP_fdr_thr, boot_w_fdrsig, w_thresh_fdr,
        dist_from_hyperplane_xval, class_probability_xval,
        crossval_accuracy, crossval_accuracy_within,
        classification_d_singleinterval, classification_d_within,
        regression_d_singleinterval, regression_d_within,
        d_singleinterval, d_within, mult_obs_within_person,
        ClassificationModel, SVMModel, SVRModel,
        trIdx, teIdx, nfolds,
        accfun_descrip, cverrfun, cverr, crossval_accuracy_descrip,
        prediction_outcome_r, prediction_outcome_r_descrip, pred_outcome_r,
        mse, rmse, meanabserr, r_squared,
        scorediff, scores_within_id, Y_within_id,
        ROC_forced_choice, ROC_single_interval, weight_obj
    end

ROUTING TABLE (the part that does the work):

    A static method `predictive_model.field_routing()` returns an N x 3
    cell array mapping every known input-struct field name to its
    {category, subfield} home. The constructor walks an input struct
    once, looks up each field name in the table, and writes to the
    right location. Unrouted fields land in legacy_extras so nothing
    is dropped silently. Adding a new wrapper = adding new rows.

    Legacy translations are encoded in the routing table directly:
        SVMModel, SVRModel, ClassificationModel  -> ml_model
        classification_d_singleinterval          -> error_metrics.d_singleinterval
        regression_d_singleinterval              -> error_metrics.d_singleinterval
        classification_d_within                  -> error_metrics.d_within
        regression_d_within                      -> error_metrics.d_within
        pred_outcome_r                           -> error_metrics.prediction_outcome_r
        models                                   -> fold_models
        bootstrap                                -> bootstrap_results
        y (lowercase)                            -> inputs.Y

PREDICATES (implemented Phase 0):
    is_fitted(obj)       -- true if any categorised fitted-state field has content
    is_classifier(obj)   -- true if task='classification' or Y is binary
    is_regressor(obj)    -- true if task='regression' or Y has >2 unique values
    clone(obj)           -- new object: same hyperparameters, empty fitted state

VERIFIED PROPERTIES OF THE LAYOUT:
    - Round-trip from a synthetic xval_SVM-style struct preserves every
      tested legacy alias (Y, yfit, w, crossval_accuracy, d_singleinterval,
      SVMModel, ClassificationModel, boot_w_ste, nfolds, modeloptions).
    - is_classifier correctly returns true for binary Y, false otherwise.
    - clone() preserves all hyperparameters and clears all fitted state.
    - validate_object accepts well-formed objects and rejects malformed
      modeloptions (the original predicate that short-circuited
      incorrectly was rewritten to actually validate odd-position names).

----------- 4c. Required and conventional methods -----------------------------

Required (every predictive_model has these):

    obj = predictive_model(varargin)
        % constructor: name/value hyperparameters OR struct (legacy)
    obj = fit(obj, X, Y, varargin)
        % train on full data; populate ml_model, Beta, intercept; record history
    [yhat, score] = predict(obj, Xnew)
        % delegate to predict(obj.ml_model, ...); accept fmri_data also via
        % a small adapter that converts to numeric and aligns voxel space
    s = score(obj, X, Y)
        % evaluate obj.scorer; higher = better
    tf = is_fitted(obj)
    tf = is_classifier(obj)
    tf = is_regressor(obj)
    new_obj = clone(obj)
        % copies hyperparameters; clears fitted state
    obj = compact(obj)
        % runs compact() on obj.ml_model and per-fold models
    obj = validate_object(obj, varargin)
        % the existing method, with the modeloptions predicate fixed

Conventional (composable analytics — return a populated predictive_model):

    obj = crossval(obj, X, Y, varargin)
        % name/value: 'cv', splitter; 'groups', id; 'scoring', scorer;
        % 'return_estimator', logical
        % populates yfit, score, cv_score, cv_results
    obj = bootstrap(obj, X, Y, varargin)
        % populates boot_Beta and derived FDR fields
    obj = permutation_test(obj, X, Y, varargin)
        % populates perm_null_score, perm_p
    obj = grid_search(obj, X, Y, paramGrid, varargin)
        % wraps OptimizeHyperparameters or hand-rolls a grid;
        % the resulting object is still a predictive_model usable inside
        % crossval (this is how nested CV is built)
    obj = calibrate(obj, X, Y, varargin)
        % Platt or isotonic calibration via cross_val_predict + a sigmoid /
        % isotonic fit; sets obj.do_calibrate = true and stores calibrator
    obj = select_features(obj, X, Y, varargin)
        % univariate or stability-selection feature pre-screen (returns a
        % predictive_model with a stored feature mask)

Reporting / visualization:

    report(obj)
        % textual summary suitable for tor_report-style output
    plot(obj)             % default dispatch: scatter (regression) or violin
                          %                   (classification)
    plot_predicted_vs_observed(obj, ...)   % existing
    confusion_matrix(obj, ...)             % existing
    confusionchart(obj, ...)               % thin wrapper around MATLAB's
    rocplot(obj, ...)                      % uses dist_from_hyperplane_xval
    montage(obj, ...)                      % build a weight statistic_image
                                           % via fmri_data adapter, then
                                           % delegate to its montage
    surface(obj, ...)                      % same idea
    weight_image(obj)                      % return a statistic_image with
                                           % Beta, Beta_p, Beta_fdr_sig

I/O:

    s = to_struct(obj)                     % canonical persistence format
    obj = predictive_model.from_struct(s)  % round-trip

Notes:
  - All methods that mutate (fit, crossval, bootstrap, calibrate, ...) push
    a one-line history entry, image_vector-style.
  - Methods that return a model return a NEW object (value semantics), with
    fitted state copied or replaced.
  - The first argument is always the object (CanlabCore convention).

----------- 4d. Algorithm registry / dispatch ---------------------------------

obj.algorithm is a short string. fit() dispatches via a registry table that
maps algorithm name -> {fit function handle, predict-postprocess, default
modeloptions}. Keeping this as a registry (rather than a big switch) means
adding a new algorithm = adding one row.

Initial registry (target):

    name              task            fit fn         notes
    'svm'             classification  fitcsvm        default kernel=linear
    'linear_svm'      classification  fitclinear     high-dim
    'logistic'        classification  fitclinear     Learner='logistic'
    'svr'             regression      fitrsvm
    'linear_svr'      regression      fitrlinear     high-dim
    'lassopcr'        regression      custom         existing CanlabCore algo
    'lassopcr_glm'    regression      custom         matlab-lasso variant
    'pcr'             regression      custom         existing default
    'ridge'           regression      ridge          or fitrlinear w/ Reg='ridge'
    'pls'             regression      plsregress     existing
    'multregress'     regression      mvregress
    'tree'            classification  fitctree
    'tree'            regression      fitrtree
    'rf'              classification  TreeBagger
    'rf'              regression      TreeBagger
    'knn'             classification  fitcknn
    'gp'              regression      fitrgp
    'ecoc'            classification  fitcecoc       multi-class wrapper
    'nnet'            classification  fitcnet
    'nnet'            regression      fitrnet
    'gam'             classification  fitcgam
    'gam'             regression      fitrgam
    'naive_bayes'     classification  fitcnb
    'multilevel_glm'  regression      custom         glmfit_multilevel
    'custom'          either          user fn handle

All of fmri_data.predict's current algorithms map onto this table.
xval_SVM and xval_SVR become thin one-liners around the 'svm' and 'svr'
entries.

----------- 4e. Cross-validation splitter objects -----------------------------

Small set of classes (or, simpler, a single splitter class with a `type`
field) that implement a uniform iterator:

    splits = split(splitter, X, Y, groups)
        % returns struct array with .trIdx, .teIdx (logical n x 1 each)

Initial types:
    kfold(k)
    stratified_kfold(k)
    group_kfold(k)               -- one group never split across folds
    stratified_group_kfold(k)    -- + stratify outcome within groups
    leave_one_group_out
    repeated_kfold(k, nrepeats)
    shuffle_split(n_splits, test_size)
    holdout(test_size)
    hvblock(h, v)                -- existing fmri_data.predict feature
    rolling(h, v, g)             -- existing fmri_data.predict feature
    custom_partition(fold_ids)   -- existing fmri_data.predict feature

Defaults:
    classification: stratified_kfold(5), stratified_group_kfold(5) if groups
    regression:     kfold(5),            group_kfold(5)            if groups

Bayesian hyperparameter optimization inside CanlabCore's grid_search must
respect the user-provided splitter (this is the bug that xval_SVM works
around today by hand-rolling everything).

----------- 4f. Scorer objects ------------------------------------------------

A scorer is a small object with:
    score(scorer, Y, yfit, scores) -> float
    greater_is_better :: logical
    needs_score :: logical            % true if it consumes continuous scores
                                      % rather than discrete predictions
    name :: char

Initial registry:
    accuracy            (class, !needs_score)
    balanced_accuracy
    f1
    roc_auc             (class, needs_score)
    average_precision
    log_loss
    brier_score
    mse                 (reg, greater_is_better=false)
    rmse
    mae
    r2                  (reg, greater_is_better=true)
    pearson_r
    cohens_d_singleinterval
    cohens_d_within     (needs groups)

Defaults:
    classification: balanced_accuracy
    regression:     r2

Picking a scorer object as the canonical interface (rather than a function
handle, which the current accfun property is) means cv_score is comparable
across folds without baked-in sign confusion, and grid_search can
argmax-optimize uniformly.

===============================================================================
5. INTEGRATION WITH EXISTING CanlabCore CODE
===============================================================================

----------- 5a. fmri_data.predict ---------------------------------------------

Current: returns [cverr, stats, optout]; stats is a flat struct.
Proposed:

  PHASE 1 (no breaking change).
    fmri_data.predict gains a 4th return value, `pm`, a predictive_model
    object built from stats + optout. The first three returns are unchanged.

  PHASE 2 (add an opt-in for the new path).
    fmri_data.predict accepts 'return_predictive_model' (or 'newapi') as a
    keyword; when set, it internally calls the new predictive_model.fit /
    crossval pipeline instead of the long internal switch statement, and
    converts the resulting predictive_model back to stats/optout for the
    legacy returns.

  PHASE 3 (deprecate the long internal switch).
    The internal algorithm-dispatch code in predict.m is removed; predict.m
    becomes a thin adapter that:
      (a) pulls X, Y, groups out of the fmri_data object,
      (b) constructs a predictive_model from the keyword args,
      (c) calls crossval / bootstrap on it,
      (d) reformats fields into the legacy stats/optout shape for return.
    This preserves every existing call site while collapsing the duplicated
    CV/bootstrap engines into one.

This phased approach is the only safe option given that fmri_data.predict
has 20+ years of callers in user scripts and example notebooks.

----------- 5b. xval_SVM / xval_SVR -------------------------------------------

Current: each is ~500 lines that hand-roll CV, optional Bayesian
optimization, optional bootstrap, optional repeated CV, plus plots and
prints. They convert to a predictive_model at the very end.

Proposed:

  PHASE 1.
    Leave them alone. They produce predictive_model objects already.
    Document that they're the legacy entry points.

  PHASE 2.
    Reimplement each as a ~30-line wrapper that:
      (1) builds a predictive_model with algorithm='svm' (or 'svr') and the
          user's modeloptions
      (2) calls crossval(pm, X, Y, 'groups', id, ...) (subject-grouped
          stratified)
      (3) optionally calls grid_search before crossval, for the
          hyperparameter-optimization branch (this gives proper nested CV
          rather than the partially-correct nested logic in the current code)
      (4) optionally calls bootstrap
      (5) returns the populated predictive_model
    User-visible behavior is identical; the internal engine is the new one.

----------- 5c. predict_test_suite, validate_object ---------------------------

predict_test_suite (a method on fmri_data) runs predict() with multiple
options and summarizes output. It naturally becomes a method that:
  - constructs multiple predictive_model objects
  - calls crossval on each
  - tabulates / compares cv_score, FDR-significant feature counts, etc.

validate_object on predictive_model should be fixed:
  - the modeloptions predicate is currently broken (it returns false on
    valid input and short-circuits to error()); replace with a proper
    cellfun check that allows mixed-type values in even positions
  - add an `is_fitted` consistency check (e.g., if Beta is set, ml_model
    should be set, and history should be non-empty)
  - keep the doverbose 'noverbose' flag

----------- 5d. region, statistic_image, montage, fmridisplay ----------------

The neuroimaging-aware adapter layer:

  obj.weight_image() returns a statistic_image built from obj.Beta with the
    bootstrap stats wired into the .p and .sig fields. This lets users
    immediately do:
        wi = weight_image(pm);
        threshold(wi, .05, 'fdr');
        montage(region(wi));
    without re-doing the FDR math.

  fmri_data.predictive_model_to_weights(obj, pm) (or a method on
    predictive_model) reconstructs the weight map back into the source
    fmri_data's voxel space (respecting removed_voxels via replace_empty).

  montage(pm) is a one-liner that calls weight_image then montage on the
    result.

These adapters live in @predictive_model/ (for the pm side) and
@fmri_data/ (for the fmri_data -> X,Y,groups extraction). The core
predictive_model class itself stays free of SPM and fmri_data imports.

===============================================================================
6. OPEN DESIGN QUESTIONS (DECISIONS NEEDED)
===============================================================================

Q1. VALUE SEMANTICS vs HANDLE CLASS.
    Existing image_vector / fmri_data are VALUE classes. predictive_model
    is currently a value class. Sklearn-style "fit returns self after
    mutation" is handle-class behavior. Recommendation: KEEP value
    semantics. fit / crossval / bootstrap return a NEW object. Trade-off:
    users have to remember to write `m = fit(m, X, Y)` not `fit(m, X, Y)`.
    Mitigation: document at the top of every method; consistent with the
    rest of CanlabCore.

Q2. SINGLE CLASS vs CLASSIFIER/REGRESSOR SUBCLASSES.
    Recommendation: single class for v1, with `task` and predicates.
    Revisit if predict_proba / margin / etc. start to feel awkwardly
    conditional.

Q3. WHERE DOES `id` LIVE?
    Two options:
      (a) groups vector passed to crossval / bootstrap / permutation_test
          via name-value, NOT stored on the object;
      (b) stored as obj.groups when fit/crossval is called, so the object
          remembers it.
    Recommendation: (a) for the API surface (matches sklearn, makes the
    object reusable across datasets), (b) as a side-effect (write
    obj.groups in crossval for diagnostic/plotting purposes, but never
    read it from there). The current `id` property stays as a legacy
    alias of obj.groups.

Q4. SHOULD predictive_model BE IMAGE-AWARE?
    Tempting: have fit() accept fmri_data directly. Avoiding: keeps the
    class testable without SPM, keeps the abstraction clean, and the
    adapter layer (fmri_data.fit_predictive_model) is a one-pager. The
    convenience of `fit(pm, fmri_data_obj, [], 'groups', id)` is mild;
    accept the small extra step.
    Recommendation: NUMERIC-ONLY in the core class. Image polymorphism
    handled by the adapter.

Q5. DEPRECATION PATH FOR LEGACY PROPERTIES.
    The current property names (w, boot_w, dist_from_hyperplane_xval,
    SVMModel, classification_d_singleinterval, etc.) are referenced by
    xval_SVM/SVR docstrings and likely by user scripts. Hard rename will
    break user code.
    Recommendation: keep them as PROPERTIES (cheap; MATLAB doesn't punish
    you for having many properties), document them as legacy aliases,
    have the new methods write both the new and legacy field on every
    fit/crossval. Plan a deprecation warning ~2 releases out, removal ~4.

Q6. WHERE DOES STANDARDIZATION HAPPEN?
    fmri_data.predict's various cv_* algorithms each handle scaling
    differently (PCR centers; lassopcr centers; svm wants -1/1 outcomes;
    fitclinear has 'Standardize' option). In sklearn this is a Pipeline
    concern.
    Recommendation: for v1, expose `'standardize',true` as a
    predictive_model hyperparameter, default true, applied identically
    inside fit and applied via stored mean/std at predict time. Add a
    `'standardize_method'` ('zscore' | 'rescale' | 'norm' | 'none') if
    we need flexibility later. Defer a full Pipeline class to v2.

Q7. PARALLELISM.
    fmri_data.predict has built-in parfor; xval_SVM uses parfor.
    Recommendation: respect `use_parallel` hyperparameter; default to
    true if a parallel pool is open or auto-start-eligible.

Q8. RANDOM STATE.
    Should be a hyperparameter, set via rng() at the start of any
    stochastic operation (fold partition, bootstrap, permutation). Sklearn
    treats this as essential for reproducibility; CanlabCore currently
    handles it inconsistently.

Q9. WHAT TO DO ABOUT THE EXISTING ALGORITHM FILES (cv_pcr, cv_lassopcr,
    cv_lassopcrmatlab, cv_svm, cv_svr, cv_pls, cv_multregress,
    cv_univregress, cv_multilevel_glm, cv_mlpcr)?
    These are pretty-well-debugged functions with their own quirks. They
    each take (xtrain, ytrain, xtest, optional_inputs) and return (yfit,
    other_outputs). They are easy to wrap as predictive_model 'algorithm'
    entries -- the registry's fit function for 'lassopcr' just calls
    cv_lassopcr internally. We do NOT need to rewrite the algorithms.
    Recommendation: WRAP, don't rewrite. The wrapper handles the CV loop;
    the algorithm file handles the per-fold fit.

Q10. THE `crossval` METHOD NAME COLLIDES WITH MATLAB BUILT-IN.
    MATLAB's classification objects have a `crossval` method that returns
    a partitioned model. Method dispatch on @predictive_model means
    calling crossval(pm, ...) on a predictive_model picks ours; but the
    name overload is mildly confusing.
    Options: keep `crossval`; rename to `cross_validate` (sklearn-style);
    rename to `xval`.
    Recommendation: KEEP `crossval` (it's already the stub name, matches
    MATLAB's verb, and dispatch by class works). Document the difference.

8. RISKS AND KNOWN TRAPS
===============================================================================

R1. Quietly breaking user scripts.
    The biggest risk is renaming a property and breaking working code in
    user labs around the world. Mitigation: legacy aliases stay; only
    additive changes in early phases; deprecation warnings before removal.

R2. Underestimating the per-algorithm quirks.
    cv_lassopcr, cv_lassopcrmatlab, cv_svm (Spider package!), cv_svr each
    have non-obvious behaviors documented only in their programmer-notes
    blocks. Wrap, don't rewrite. Read each one's input/output contract
    before writing its registry row. Note that cv_svm uses the legacy
    Spider package; consider routing 'svm' to fitcsvm in the new path and
    keeping cv_svm only as a legacy-path option.

R3. The MATLAB compact() trap.
    fitcsvm on full-brain data carries copies of X and SupportVectors
    that can hit hundreds of MB. The new pipeline MUST call compact() on
    every per-fold model and on ml_model before storing. Forgetting this
    will cause saved predictive_model objects to balloon.

R4. removed_voxels / replace_empty.
    The fmri_data adapter must respect the source object's removed_voxels
    when reconstructing weight maps. This is the classic CanlabCore trap
    (see CLAUDE.md). Build a small unit test that fits on data with
    removed voxels and confirms the weight_image has zeros in the right
    places.

R5. Subject grouping subtleties.
    Stratification within groups (the right thing for binary classification
    with repeated measures) requires stratified_group_kfold, not just
    group_kfold. xval_SVM does this correctly today; the new splitter
    must as well.

R6. Hyperparameter optimization with subject-grouped CV.
    OptimizeHyperparameters defaults to non-grouped CV. The new path
    needs to construct a custom 'CVPartition' that respects groups and
    pass it through, or hand-roll bayesopt. Easy to get subtly wrong.

R7. The validate_object modeloptions predicate.
    The current implementation is wrong but never errors because the
    short-circuit clause covers the bad branch. Anyone reading the
    code thinks it works. Replace it cleanly and add a test that
    deliberately passes a malformed modeloptions cell.

R8. Method-name conflict with built-ins.
    `crossval`, `train`, `predict`, `score`, `loss`, `montage` all
    overload either MATLAB built-ins or other class methods. Dispatch
    by class handles it correctly, but be alert in mixed scripts.

R9. Random state and bootstrap reproducibility.
    Today, multiple sites parallelize bootstrap by re-seeding with
    `rng('shuffle')` per worker. The new path needs a documented
    convention (master seed at fit; per-worker offset by worker id)
    so users can either reproduce exactly or run independently.
